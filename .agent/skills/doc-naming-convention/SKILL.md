# Goal

统一 `docs/`、`doctor/paper1/docs/` 目录及其子目录下所有方案、设计报告和投稿附件的文档命名规范，强制添加版本号和时间戳。

## Instructions

1. **识别触发条件**：
   - 检查目标路径是否为 `docs/`、`doctor/paper1/docs/` 或其子目录（如 `docs-zh/`、`doctor/paper1/docs/study/` 等）。
   - 检查要输出的文档内容类型是否属于以下任一类别：
     - **方案 / 设计报告**：系统架构设计、技术方案、实验方案等。
     - **投稿附件**：投稿信（Cover Letter）、审稿回复信（Response Letter）、作者声明等。
     - **论文稿件**：已由 `build_pdf_versioned.sh` 独立管理，本技能仅规定命名格式供参考。

2. **格式化文档名称**：

   **方案 / 设计报告**（Markdown）：
   - 格式：`v<版本号>_<YYYYMMDD>_<文档简短描述>.md`
   - 示例：`v1.0_20260612_系统架构设计方案.md`

   **投稿附件**（PDF，由 LaTeX 编译生成）：
   - 格式：`v<版本号>-<文档类型>-<YYYYMMDD-HHMMSS>.pdf`
   - `版本号`：与当前论文稿件版本号保持一致（从 `docs/` 下已有 `v*-论文稿件-*` 文件自动检测最大版本号）。
   - `文档类型`：使用固定短名称，如 `投稿信`、`审稿回复信`、`作者声明`。
   - `时间戳`：`YYYYMMDD-HHMMSS` 格式，与论文稿件版本化输出保持风格统一。
   - 示例：`v30-投稿信-20260612-095500.pdf`

   **论文稿件**（PDF，参考）：
   - 格式：`v<版本号>-论文稿件-<模板>-<YYYYMMDD-HHMMSS>.pdf`
   - 由 `scripts/build_pdf_versioned.sh` 自动管理，本技能不重复执行。
   - 示例：`v30-论文稿件-jos-20260612-085448.pdf`

3. **版本号规则**：
   - **方案 / 设计报告**：初始默认 `1.0`，更新时递增小版本（如 `v1.1`）。
   - **投稿附件 / 论文稿件**：使用整数递增版本号（如 `v30`、`v31`），从 `docs/` 目录已有文件自动检测。检测命令：
     ```bash
     ls -1 docs/ | grep -oP '^v\K[0-9]+(?=-)' | sort -n | tail -1
     ```
   - 如果用户明确指定了版本号，以用户指定的为准。

4. **静默修正**：
   - 如果用户口语中提供了不规范的文件名（如 `投稿信.pdf`），请自动将其转化为上述规范形式再执行保存。

5. **内部文档元数据对齐**：
   - Markdown 文档：确保一级标题 `#` 或文档开头包含对应的版本号和日期信息。
   - PDF 文档：LaTeX 源码中的日期字段应与输出文件名中的时间戳一致。

## Examples

| 场景 | 输出文件名 |
|------|-----------|
| 初次输出系统方案 | `docs/v1.0_20260612_系统架构设计方案.md` |
| 投稿信第 30 版 | `docs/v30-投稿信-20260612-095500.pdf` |
| 审稿回复信第 31 版 | `docs/v31-审稿回复信-20260615-140000.pdf` |
| 论文稿件 JOS 版 | `docs/v30-论文稿件-jos-20260612-085448.pdf` |

具体示例请参考 `examples/` 目录下的案例文件。

## Constraints

- 此规范适用于方案、设计报告和投稿附件类文档。
- 不得应用于常规说明（如 `README.md`、`index.rst`、开源协议）、学术论文 LaTeX 源文件、配置文件及代码文件。
- 文件名中的分隔符（下划线 `_` 用于方案，连字符 `-` 用于 PDF 附件）不可省略。
- 投稿附件的版本号必须与同期论文稿件版本号一致，避免版本不对应。
- `SKILL.md` 内容必须使用中文编写。
