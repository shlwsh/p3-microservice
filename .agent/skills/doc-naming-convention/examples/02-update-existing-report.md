# 示例 2：更新已存在的设计报告

## 场景说明
用户要求基于现有的架构方案报告进行更新修改。假设原文件名为 `v1.0_20260510_系统架构设计方案.md`。

## 输入请求
"请对现有的系统架构设计方案添加一部分关于安全性模块的设计，然后保存更新的报告到 docs 目录。"

## 执行过程
1. 检测到输出路径为 `docs`，且输出类型为“架构设计方案”（属于方案/设计报告），触发 `doc-naming-convention` 技能。
2. 识别到这是一个针对旧版本文档（`v1.0_20260510_系统架构设计方案.md`）的更新。
3. 自动将版本号递增，如设为 `v1.1`。
4. 提取当前系统日期，假设为 `20260611`。
5. 保持文档简短描述不变：`系统架构设计方案`。
6. 自动组装输出文件名为：`docs/v1.1_20260611_系统架构设计方案.md`。

## 预期输出（文档内容头部）

```markdown
# 系统架构设计方案

**版本：** v1.1
**更新日期：** 2026年6月11日

## 版本历史
- v1.1 (2026-06-11): 增加了安全性模块的设计方案。
- v1.0 (2026-05-10): 初始草案。

## 1. 概述
本文档描述了系统的整体架构设计...
```
