# 基准论文对照与改稿闭环

> **用途**：当用户要求以《软件学报》编辑视角对 p3 稿件进行重大修改、二次评审或与基准论文对比时，必须使用本清单。  
> **基准论文**：A：包航宇等《智能运维的实践:现状与标准化》（2023）；B：贾统等《基于日志数据的分布式软件系统故障诊断综述》（2020）。

---

## 1. 八维度对照门槛

评审报告与改稿方案至少覆盖以下 8 个维度，每个维度必须给出：`差距`、`修改动作`、`验收标准`、`涉及文件`。

| 维度 | 对照重点 | 主责角色 | 稿件验收标准 |
|------|----------|----------|--------------|
| 结构与逻辑 | 论文 A/B 均有清晰框架；p3 需把三层采集链路提炼为框架 | ✍️ 🎓 | 摘要/引言/系统节均出现“分布式定向日志采集框架”及三层输入输出 |
| 文献综述与定位 | A 覆盖 AIOps 标准化，B 覆盖日志诊断技术框架 | 🔬 | `02_related.tex` 单独讨论 AIOps 标准化与日志诊断，表格含本文互补定位 |
| 技术创新 | A 有 AIOps-OSA，B 有四技术框架；p3 需凝练为可复用采集前端框架 | 🔍 🔬 | 三贡献在摘要、引言、结论一致；算法节含复杂度/正确性说明 |
| 实验严谨性 | A/B 的证据规模强；p3 原型规模需承认边界并给扩展计划 | 📊 🔍 | 实验节含“效度威胁与可复现性”；明确 Promtail/OTel/eBPF 基线计划 |
| 写作与呈现 | 摘要、图表、术语需达到学报标准 | ✍️ | 英文 Abstract 术语统一；关键图表为 PDF/矢量；长句拆分 |
| 实用性与标准化 | A 强调标准化与实践能力建设 | 🔬 📐 | 实现或讨论节说明 AIOps-OSA 数据采集/场景实现能力映射 |
| 算法与实现深度 | B 详述算法分类；p3 需补实现细节 | 🔍 | URL 泛化、Redis 数据模型、匹配效率、缓存/退避边界清楚 |
| 未来工作 | A/B 展望开放挑战 | 🎓 | 结论列出真实负载、多规模、ML 故障诊断、隐私保护、开源工件 |

---

## 2. 改稿闭环要求

当用户要求“根据评审意见改稿/实现改进/二次评审”时，执行以下闭环：

1. **方案先落盘**：在 `docs/` 下生成带时间戳的改进方案，命名形如 `YYYYMMDD-HHMMSS-重大修改改进方案.md`。
2. **修改主稿**：同步修改 `latex/main-zh.tex`、`latex/main-jos.tex`、`latex/sections/zh/*.tex` 与 `docs/v4-论文稿件.md`。涉及数值、贡献或术语时必须双稿同步。
3. **版本化编译**：每次成功编译后，将生成 PDF 复制为带版本号与时间戳的文件，形如 `docs/v5-论文稿件-jos-YYYYMMDD-HHMMSS.pdf`；历史 PDF 不删除。
4. **二次评审**：对修改后稿件执行轻量 Phase 0–5 回归，至少输出 `docs/YYYYMMDD-HHMMSS-二次评审报告.md`，含上一轮问题关闭情况与剩余风险。
5. **回归验证**：记录编译命令、PDF 路径、`git diff --check` 结果；若未能编译或验证，必须在二次评审报告中说明原因。

---

## 3. 二次评审重点

二次评审不重复生成完整 `reviews/{run_id}/` 时，最少检查以下项：

| ID | 检查项 | 通过标准 |
|----|--------|----------|
| R1 | 八维度建议是否落稿 | 每维至少 1 个正文或方案响应 |
| R2 | 基准论文 A/B 互补定位 | 正文明确 A 为标准化/实践框架，B 为后端诊断综述，本文为采集前端减量 |
| R3 | 实验限制是否诚实 | 未新增实测时不得声称已完成 32/64 节点、Promtail、DeathStarBench |
| R4 | 表格不溢出 | 新增/扩展表格用 `\resizebox{\textwidth}` 或合理 `p{}` 列宽 |
| R5 | 版本化文稿 | `docs/` 中存在本轮带版本号 PDF |
| R6 | 历史保留 | 不删除旧 `v4` 或旧时间戳文稿 |

---

## 4. Cover Letter 对应关系

重大修改完成后，建议生成或更新 cover letter，至少包含：

- 对编辑 8 个维度建议的逐项回应；
- 新增/修改的章节和图表位置；
- 未完成大规模真实负载实验的原因与计划；
- 代码、数据、Docker Compose 复现工件的公开计划；
- 与论文 A/B 的互补关系：AIOps 标准化框架中的数据采集能力、日志故障诊断链路中的前端减量环节。
