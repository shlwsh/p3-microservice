# 计算机科学顶级期刊论文规范指南与评审要求

> **用途**：作为 Phase 1–5 的通用高水平期刊门槛，适用于《软件学报》《计算机学报》及同等级中文计算机类期刊。  
> **关系**：本文是通用规范；JOS 细则仍以 [软件学报投稿要点.md](软件学报投稿要点.md) 为准，文献归档以 [参考文献归档细则.md](参考文献归档细则.md) 为准。

---

## 1. 稿件评审要求

| 维度 | 评审问题 | 通过标准 | 常见 Issue |
|------|----------|----------|------------|
| 选题价值 | 是否直击真实痛点，而非工程展示？ | 引言明确痛点、应用背景、未解决挑战 | `TOPIC-VALUE` |
| 创新性 | 贡献是否可被审稿人复述？ | 引言末条目化列出 3–4 点核心贡献 | `TOPIC-CONTRIB` |
| RQ 导向 | 实验是否围绕 RQ/SQ 组织？ | 实验节有 RQ/SQ 与指标/图表映射 | `EXP-RQ` |
| 基准与数据 | 是否使用真实或公认数据/系统？ | 有真实负载、公开基准或诚实说明原型边界 | `EXP-DATA` |
| 消融与参数 | 是否验证核心组件必要性？ | 有消融、参数敏感性或明确后续计划 | `EXP-ABL` |
| 方法严密 | 是否有形式化、伪代码、推导和架构图？ | 至少核心算法有伪代码/复杂度/边界条件 | `METHOD-FORMAL` |
| 可复现 | 是否给出代码、配置、数据或开源计划？ | 有脚本、Docker/配置、数据路径或开源承诺 | `REPRO` |

## 2. 文档结构与篇幅规范

### 2.1 首页与元数据

| 项 | 要求 | 主责 |
|----|------|------|
| 中英文标题 | 中文题名与英文题名语义一致 | ✍️ |
| 中英文摘要 | 摘要包含背景、挑战、方法、结果四要素 | ✍️ |
| 关键词 | 中英关键词数量与术语一致 | ✍️ |
| 作者机构 | 作者、单位、邮箱、通讯作者准确 | ✍️ |
| 基金/声明 | 无基金不得留占位；AI 辅助声明规范 | ✍️ 📐 |
| 引用格式 | 若期刊要求，首页或脚注给出标准引用格式 | ✍️ |
| 作者简介 | 文末含学历、研究方向，必要时含 CCF/学会会员级别 | ✍️ |

### 2.2 正文结构建议

| 章节 | 建议篇幅 | 内容要求 |
|------|----------|----------|
| 引言与背景 | 约 10% | 漏斗式引出痛点，末尾条目化贡献 |
| 相关工作与数据基座 | 约 15% | 分类综述、数据特征、同类对比 |
| 方法设计/核心技术 | 约 35% | 架构图、算法框图、形式化、实现边界 |
| 实验设置与深度分析 | 约 25% | RQ、数据集、指标、基线、消融、有效性威胁 |
| 工程实践/总结展望 | 约 15% | 部署、标准化、局限、未来工作 |

> 对 p3 工程稿：若未达到长篇综述的 8000–10000 字，也必须保证“方法 + 实验 + 局限”有足够厚度，不能呈现为项目总结。

## 3. 行文表达规范

| 场景 | 要求 | 反模式 |
|------|------|--------|
| 整体语体 | 客观、书面、少第一人称；使用“本文提出/实验表明” | “我们做了一个系统” |
| 摘要 | 背景→然而痛点→方法→量化结果 | 只写系统功能，无结果 |
| 引言 | 宏观到微观，最后条目化贡献 | 背景散乱，贡献藏在段落中 |
| 相关工作 | 客观评价局限，服务本文定位 | 文献罗列，无对比结论 |
| 方法 | 首先/随后/最后引导，变量和假设清楚 | 仅流程描述，无形式化 |
| 实验 | 不只报数，还解释“为什么” | 只说优于基线，无原因分析 |
| 总结 | 坦陈局限，给出具体前瞻 | “未来进一步优化”空泛 |

## 4. 图表与公式规范

| 类型 | 规范 | 审核动作 |
|------|------|----------|
| 公式 | 独立成行，右侧连续编号，如 `(1)` | 检查 `equation` 或编号形式 |
| 变量 | 变量斜体，公式后紧跟“其中，”逐一解释 | 查变量是否未定义 |
| 表格 | 表题在表上方，三线表，最优值加粗 | 查 `\caption`、`\toprule`、`\textbf{}` |
| 图片 | 图题在图下方，正文引用，子图用 (a)(b) | 查 `figure`、`\caption`、`\ref` |
| 数据图 | 坐标轴、单位、误差条/σ 或说明 | 查图表是否可复算 |
| 架构图 | 必须解释模块输入/输出与数据流 | 查正文是否逐模块说明 |
| 消融图 | 必须标明实测/仿真/解析模型 | 防止混淆证据等级 |

## 5. 参考文献规范

| 项 | 要求 | 审核动作 |
|----|------|----------|
| 格式合法性 | `references.bib` 的 BibTeX 语法合法、键名唯一且无冲突 | 编译检查或运行 `bibtex` 是否报语法致命错误 |
| 引用完全解析 | 正文所有 `\cite{}` 成功解析，无 `[?]` 占位符 | 检查编译日志是否包含 `Undefined references` 警告，PDF 中是否存在 `[?]` 标识 |
| 元数据来源 | 参考文献元数据须从学术平台（如 Google Scholar, dblp, CrossRef 等）以标准 BibTeX 格式导入，字段完整 | 抽查新近添加的文献，检查 `year`/`author`/`journal` 等字段是否完整，避免潦草手工填写 |
| 正文引用 | 数字方括号体例由模板控制；连续引用应合并 | 编译检查引用格式 |
| 中文文献英译 | 中文文献必要时提供英文题名或 `in Chinese` 标注 | 检查 bib 字段或参考文献输出 |
| 类型标识 | 期刊/会议/学位/在线资源格式统一 | 抽查 `.bbl` 或 PDF |
| DOI/URL | 每条 cited 可核实 | 跑 `verify_cited_papers.py --download` |
| 中英文平衡 | JOS 场景中文文献 ≥30% 或至少 8 篇 | 统计 bib 和正文引用 |
| 新近与高影响 | 近 5 年前沿与经典文献兼顾 | 查 2023+ 条目和核心综述 |
| 归档 | PDF 或官网/CrossRef 快照进入 `data/papers/` | manifest `paywall=0, failed=0` |

**中文著录建议**：若 BibTeX 输出不支持中英双语对照，至少在正文相关工作中给出中文作者和中文题名，参考文献条目保留 DOI/URL；终稿可按期刊模板补英文译名与 `[J]/[C]/[OL]` 类型标识。

## 6. 顶级期刊快速裁决规则

| 等级 | 条件 |
|------|------|
| 可接收/小修 | 贡献清晰，实验支撑充分，文献可核，结构与图表规范，无 P0 |
| 大修 | 工程实现扎实，但实验规模、基线、文献、结构或规范存在明显缺口 |
| 拒稿倾向 | 缺创新、缺真实实验、文献不可核、格式严重不合规或学术诚信风险 |

**PI 裁决要求**：若 `TOPIC-CONTRIB`、`EXP-DATA`、`METHOD-FORMAL`、`C6`、`J2/J3` 任一为 P0，默认不建议直接接收。
