# 16/32 节点实验路线与执行方案

**日期**：2026-06-11  
**目标**：在现有 8 节点证据基础上，补充更高规模的真实集群验证，并规划 50 万级压力模拟，提升论文实验说服力，同时避免将模拟数字误写为真实实测。

---

## 一、实验路线

本轮实验分三层推进：

1. **16 节点真实 Docker 集群验证（本轮立即执行）**
   - 目标：将论文主实验从 8 节点扩展到 16 节点真实集群口径。
   - 脚本：`scripts/p3_run_phase4.sh 16`
   - 产物：`experiments/results/phase4/phase4_latest.json` 与时间戳 JSON。
   - 证据等级：真实 WSL/Docker 集群实测。

2. **32 节点真实 Docker 集群试跑（16 节点稳定后执行）**
   - 目标：验证当前机器在 32 节点下的可运行性与趋势。
   - 建议先使用短时压测，确认容器稳定、网关路由与 Center 指标正常，再决定是否进入论文主表。
   - 证据等级：若完整跑通，则作为真实集群扩展证据；若资源波动明显，则仅作为可行性观察。

3. **50 万级日志压力模拟（后续脚本扩展后执行）**
   - 目标：回答“全量基线 50 万+ 入库”在大日志量下是否仍保持减量优势。
   - 方法：基于 Phase 5 多进程模拟器扩展 `nodes`、`duration`、`emit_interval_ms` 或 `target_full_logs` 参数。
   - 证据等级：基于真实分布的压力模拟，不替代真实 Docker 集群实测。

---

## 二、本轮 16 节点验证内容

执行命令：

```bash
scripts/p3_run_phase4.sh 16
```

该脚本将执行以下步骤：

1. 生成 16 节点 Docker Compose 与 Nginx 路由配置。
2. 重建 16 个 `httpbin` 服务及对应 16 个 Sidecar Agent。
3. 运行 Phase 4 基准：
   - 8/16 节点定向模式规模矩阵；
   - 规则级漏报探针；
   - HTTP 到 Center 的端到端延迟；
   - 生产迹风格混合负载；
   - 稳态漏报探针；
   - Center stop/start 恢复测试；
   - Promtail/OTel/eBPF 同负载估算基线。
4. 更新 Phase 4 图表。

## 三、判定标准

16 节点结果进入论文的最低条件：

1. `phase4_latest.json` 成功生成，且 `scale.16` 包含 3 次重复运行。
2. 16 节点 `loki_stored_mean` 随节点数增长但不出现异常爆炸。
3. `attention_list_items` 与高价值 URL 模式覆盖正常。
4. 端到端延迟和稳态漏报指标可解释；若探针命中率受批处理影响，正文必须保留限制说明。
5. Docker 容器无大面积重启或 OOM。

---

## 四、论文写入原则

1. **真实集群实测数字**可进入摘要、实验主表和结论。
2. **32/64 节点外推**只能写为趋势参考，不作为主结论。
3. **50 万级压力模拟**只能写为“压力模拟”或“基于真实分布的多进程模拟”，不得表述为 Docker 集群实测。
4. 若 16 节点结果与 8 节点趋势一致，优先在摘要中写“8/16 节点真实集群验证”，而不是简单替换为单个更大的数字。

---

## 五、风险与缓解

| 风险 | 表现 | 缓解 |
|---|---|---|
| Docker 资源波动 | 容器启动慢、请求超时、指标抖动 | 先完成 16 节点；32 节点先短时试跑 |
| 高负载改变实验口径 | 50 万级数字不可与 180s/并发 50 主实验直接比较 | 单列压力实验，不替换主实验 |
| 探针即时命中率偏低 | Agent 2s 批处理窗口影响 received 增量 | 使用规则级漏报与稳态说明区分 |
| 工业基线证据不足 | Promtail/OTel/eBPF 未完整端到端部署 | 明确估算/模拟/实测三类证据等级 |
