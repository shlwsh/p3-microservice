实验口径：32 节点，定向模式，60 s，重复 3 次。

| 轮次 | Loki 入库增量 | Gateway Redis 增量 | 关注清单项 | Agent 平均 CPU | Center CPU |


该总量包含重建后 warmup、压测和历史状态合并影响，论文引用时应以时间戳 JSON 中 `scale.32.runs` 的每轮增量为准。

## 三、结论

1. **32 节点真实 Docker 集群可运行**：当前 WSL/Docker 资源可以支撑 32 节点原型规模验证。
2. **定向入库量仍受控**：每 60 s 约 6700 条网关流量日志下，Loki 入库稳定为 99 条/轮。
3. **关注清单达到 Top-K 上限**：32 节点下关注清单项为 50，说明路由模式数量已触达当前 `Top-K=50` 配置上限。
4. **资源开销仍低**：Agent 平均 CPU 约 0.06%，Center CPU 均值约 0.32%，未观察到明显瓶颈。

## 四、论文写入建议

可将该结果作为 32 节点真实集群规模扩展证据写入“规模扩展与质量指标”小节：

> 在 32 节点真实 Docker 集群下，定向模式 60 s 规模验证产生 `6700.67±55.41` 条网关流量日志，Loki 入库稳定为 `99.0±0.0` 条，关注清单达到 `Top-K=50` 上限，Agent 平均 CPU 为 `0.06±0.01%`，Center CPU 为 `0.32±0.40%`。

建议谨慎处理两个点：

1. 32 节点结果只跑了 scale-only，没有重新执行漏报率、端到端延迟和恢复测试；正文中不要写成“完整质量验证”。
2. 关注清单达到 50 项上限，后续若要继续推进 64 节点，应同步评估 `Top-K` 参数对覆盖率的影响。
