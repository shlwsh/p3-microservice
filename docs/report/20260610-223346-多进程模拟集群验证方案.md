# 多进程模拟集群验证方案

**目的**：针对审稿人提出的三项关键实验缺口，通过多进程模拟方式在本地环境补充验证证据  
**日期**：2026-06-10  
**依赖**：`experiments/results/phase3/phase3_latest.json`（三期实测原始数据）

---

## 一、背景与目标

审稿人预审报告指出三项高优先级实验缺口：

| 编号 | 缺口 | 审稿人 | 当前状态 |
| ---- | ---- | ------ | -------- |
| E1 | 同频率对照（2s vs 2s） | R1/R2/综合 P0 | 理论分解，无实测 |
| E2 | Promtail 同集群实测 | R1/综合 P1 | 仅估算 |
| E3 | 集群实测消融 | R1/综合 P1 | 仅策略仿真 |

**验证策略**：由于当前无法启动完整 Docker 集群（WSL/资源限制），采用**多进程模拟**方式：

- 基于三期实测原始数据（`phase3_latest.json`）中的真实分布参数
- 用 Python 多进程模拟 8 节点 Agent 的日志产生与过滤行为
- 模拟器复现核心算法逻辑（关注清单匹配、固定缓存块、退避机制）
- 输出与三期实测格式一致的 JSON 结果，可直接对比

## 二、模拟器架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Orchestrator (主进程)                      │
│  - 生成混合 HTTP 请求流（正常/500/慢请求/其他）                  │
│  - 管理关注清单（Top-K URL 模式）                              │
│  - 收集各 Agent 上报日志                                      │
│  - 统计 Loki 入库量、CPU 模拟开销                              │
└────────────┬────────────────────────────────────────────────┘
             │ multiprocessing.Queue
    ┌────────┼────────┬────────┬──── ... ────┐
    │        │        │        │              │
┌───┴──┐ ┌───┴──┐ ┌───┴──┐ ┌───┴──┐    ┌────┴───┐
│Agent1│ │Agent2│ │Agent3│ │Agent4│    │Agent8  │
│(进程)│ │(进程)│ │(进程)│ │(进程)│    │(进程)  │
└──────┘ └──────┘ └──────┘ └──────┘    └────────┘
  每个 Agent 进程：
  - 按配置的发射间隔产生 Apache Combined 风格日志
  - 根据关注清单匹配规则过滤
  - 固定缓存块管理 + 退避重试模拟
  - 通过 Queue 将匹配日志上报 Center
```

### 2.1 关键参数

| 参数 | 说明 | E1 同频率对照 | E2 Promtail 模拟 | E3 消融 |
| ---- | ---- | ------------ | ---------------- | ------- |
| `emit_interval_ms` | 日志发射间隔 | 2000 (both) | 400 (全量) | 2000 |
| `filter_mode` | 过滤模式 | directed / full-same-freq | promtail-static | 见下表 |
| `duration_sec` | 压测时长 | 180 | 180 | 180 |
| `n_agents` | Agent 数量 | 8 | 8 | 8 |
| `repeats` | 重复次数 | 3 | 3 | 3 |

### 2.2 消融实验矩阵 (E3)

| 配置名 | 关注清单 | 二次过滤 | 缓存块 | 退避 | 说明 |
| ------ | -------- | -------- | ------ | ---- | ---- |
| baseline | ✅ | ✅ | ✅ | ✅ | 定向完整模式 |
| no-attention | ❌ | ✅ | ✅ | ✅ | 无清单 → 全部上报 |
| no-secondary | ✅ | ❌ | ✅ | ✅ | 关闭中心二次过滤 |
| no-cache | ✅ | ✅ | ❌ | ✅ | 无缓存块 → 逐条上传 |
| no-backoff | ✅ | ✅ | ✅ | ❌ | 固定重试间隔 |

## 三、实验实现细节

### 3.1 E1: 同频率对照

**核心变量隔离**：将全量模式的发射间隔从 400ms 改为 2000ms（与定向模式相同），其余参数不变。

- **定向模式 (2s)**：基于关注清单匹配，仅上报匹配日志 → 预期入库 ~72 条（与三期实测一致）
- **全量模式 (2s)**：关闭所有过滤规则，上报全部日志 → 预期入库 ~4388/5 ≈ 878 条
- **预期结果**：同频率降幅 = 1 - 72/878 ≈ 91.8%，验证策略过滤的独立贡献

### 3.2 E2: Promtail 模拟

模拟 Promtail 静态标签过滤规则：
- 保留所有 `status >= 400` 的日志
- 保留所有 URL 包含 `/delay/` 的日志
- 丢弃其余日志

与定向模式对比：Promtail 静态规则无动态 Top-K 清单，保留比例固定。

### 3.3 E3: 集群消融

在模拟器中逐一关闭组件，测量 Loki 入库量变化：
- 关闭关注清单 → 验证清单匹配的减量贡献
- 关闭二次过滤 → 验证中心过滤的去重贡献
- 关闭缓存块 → 验证缓存对上传效率的影响
- 关闭退避 → 验证退避对传输可靠性的影响

## 四、日志产生模型

基于三期实测数据的请求分布：
- 每轮 180s 压测，网关流量约 19,700 条（Redis 增量均值）
- 关注清单稳定 16 条 URL 模式
- 请求分布：~60% 正常 / ~20% 5xx / ~12% 慢请求 / ~8% 其他
- 定向模式匹配入库 72 条/轮
- 全量模式（400ms 发射）入库 4388 条/轮

每个 Agent 进程按照上述分布独立产生日志行，8 个 Agent 并行运行。

## 六、验证标准

| 实验 | 预期结果 | 验证通过条件 |
| ---- | -------- | ------------ |
| E1 同频率 | 定向 ~72, 全量(2s) ~878 | 策略降幅 > 85% |
| E2 Promtail | 定向 < Promtail < 全量 | 定向入库量最低 |
| E3 消融-无清单 | 入库量接近全量 | 清单贡献 > 90% |
| E3 消融-无二次过滤 | 入库量略增 | 二次过滤贡献 < 10% |
| E3 消融-无缓存 | 入库量不变，上传次数增加 | 缓存影响传输效率不影响入库 |
