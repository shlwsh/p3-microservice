## 2. 研究背景

微服务把单体系统拆成多个独立服务，日志也随之分散到网关、业务容器和 Sidecar 中。全量采集虽然信息完整，但会把大量低价值 INFO 日志推入中心端，造成节点资源、网络带宽、Loki 存储与索引压力。

本文的思路是把采集决策前移：网关先观察哪些请求慢、错或异常，Center 据此生成关注清单，再让服务侧 Agent 只上传与高价值请求相关的应用日志。

## 4. 三条贡献

1. **框架贡献**：提出网关预筛选、节点定向采集、中心二次过滤三层组成的分布式定向日志采集框架。
2. **算法贡献**：给出关注清单动态生成、三次策略转换、固定缓存块和压力感知指数退避的形式化描述与 Go 实现。
3. **验证贡献**：在 WSL/Docker 八节点原型上完成端到端验证、三期重复对比实验、策略仿真消融与算法微基准测试。

## 5. 本文定位

| 对照方向 | 本文关系 |
|----------|----------|
| AIOps 标准化 | 支撑数据采集能力，降低进入 AIOps 平台的数据规模 |
| 日志故障诊断 | 位于后端解析、异常检测和根因定位之前，减少输入规模 |
| OpenTelemetry 采样 | OTel 面向 Trace/span；本文面向 URL 模式和应用日志 |
| eBPF 采集 | eBPF 偏基础设施事件；本文强调网关策略驱动的应用日志采集 |
