## 2. 研究背景（大白话）

微服务一拆多，日志也跟着「散、多、杂」。传统 ELK/Loki **全量采集**像 24 小时上传所有监控录像——带宽、存储和排障成本都高。

我们的思路：**先看网关流量谁有问题**（慢响应、错误码、热点 URL），再让各服务 **只采这些请求相关的应用日志**，其余不采或少采。

## 3. 科学问题

|------|------|---------|

| SQ2 | 固定缓存 + 退避能否在弱网/故障下可靠？ | A2/A3 消融 + R1/R2 |
| SQ3 | 三次策略转换能否贯通全链路？ | 架构 + E2E 演示 |

## 4. 四项要点

1. **架构**：主从式定向采集（OpenResty 网关 + Center + Sidecar Agent），Sidecar 可嵌入微服务。
2. **算法**：关注清单动态生成、固定缓存块、定向策略三次转换、压力感知指数退避。
3. **验证**：WSL/Docker 原型上完成首期仿真 → 二期实测 → **三期八节点长时压测**，Loki 入库降幅 **98.4%**（见 L1 [`验证结果_三期.md`](../验证结果_三期.md)）。
4. **文献与投稿门禁**：v6 正文唯一引用 **73** 篇（中文 30、英文 43），全部通过 `data/papers/cited_papers_manifest.json` 归档核验。

## 5. 本文不做

- 不做 K8s 生产级多租户运维（原型为 Docker Compose）
- 不做 LLM 智能日志分析 / 根因自动定位
- 不做与商业 APM（Datadog 等）全功能对标
- 不做跨语言二进制协议的全量适配
