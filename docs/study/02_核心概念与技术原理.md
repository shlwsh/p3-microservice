# 02 · 核心概念与技术原理

## 1. 定向采集 vs 全量采集

| 维度 | 全量采集 | 定向采集 |
|------|----------|----------|
| 采集对象 | 所有应用日志 | 与关注清单匹配的日志，ERROR 兜底 |
| 策略来源 | 静态配置或标签 | 网关 access log 动态生成 |
| 减量位置 | 多在存储或查询端 | 采集前端，即 Sidecar 上传前 |
| 资源开销 | 上传和入库压力高 | 低上传量、低 Loki 入库量 |
| 风险 | 成本高、噪声多 | 可能丢弃非匹配 INFO，需要声明 trade-off |

核心 trade-off：本文牺牲部分低价值日志完整性，换取可解释、可审计的采集前端减量。

## 2. 三层框架

```text
网关预筛选        节点定向采集                 中心二次过滤
OpenResty/Lua -> Center/Redis/关注清单 -> Agent匹配/缓存 -> Loki
```

| 层 | 输入 | 输出 | 作用 |
|----|------|------|------|
| 网关预筛选 | URL、状态码、响应时延 | 高价值流量候选 | 从入口识别慢请求、错误请求和异常 URL |
| 节点定向采集 | 关注清单、本地应用日志 | 命中日志批次 | 在 Sidecar 内完成上传前过滤 |
| 中心二次过滤 | 上传批次、清单版本、去重状态 | Loki 入库日志 | 统一版本、TTL、去重和标签 |

## 3. 关注清单

关注清单是本文的动态策略载体，来自网关流量而不是静态文件。

生成流程：

1. 从 Redis 窗口读取网关日志。
2. 过滤慢请求、5xx、异常高价值记录。
3. URL 泛化：数字段、UUID、长哈希归一为模式。
4. 按出现次数和严重度加权，取 Top-K。
5. 附加版本号与 TTL，通过 gRPC 下发给 Agent。

## 4. 三次策略转换

| 次序 | 输入 | 输出 | 作用 |
|------|------|------|------|
| 第一次 | 定向策略 S | 网关采集规则 | 控制网关侧预筛阈值 |
| 第二次 | 网关日志 + S | Agent 过滤规则 | 生成节点可执行关注清单 |
| 第三次 | 关注清单 + 服务日志 | Loki 存储规则 | 中心侧版本、TTL、去重和入库 |

正确性依赖：同一关注清单版本、同一 URL 泛化函数、未过期 TTL。字段缺失或格式不兼容时退化为 ERROR 级别兜底。

## 5. 固定缓存块与退避

每个 Agent 用固定上限缓存块控制内存占用；上传失败时使用压力感知指数退避：

```text
d_n = min(d_max, d_0 * rho^n + rand(0, d_0 * rho^n * xi))
```

v9 LaTeX 已将该公式编号为 `eq:backoff`，参数为：`d0=200 ms`、`rho=2.0`、`dmax=30 s`、`xi=0.3`、最大重试 6 次。中心不可达时写入 BoltDB 本地兜底。

## 6. 与 AIOps/日志诊断的关系

| 方向 | 代表对象 | 本文位置 |
|------|----------|----------|
| AIOps 标准化 | 包航宇等 AIOps-OSA | 数据采集能力的前端减量组件 |
| 日志故障诊断 | 贾统等日志诊断综述 | 后端诊断之前的输入缩减 |
| Loki/Promtail | 工业日志栈 | Loki 保留，Promtail 式全量上传策略被动态清单替代 |
| OpenTelemetry | Trace/span 采样 | 与 URL 粒度应用日志定向采集互补 |
| eBPF | 内核/系统事件 | 与应用层语义采集互补 |

## 7. 当前数字口径

- 主实验：8 节点，180 s，并发 50，每模式 3 轮。
- 主结果：Loki 入库 72 条 vs 4388±1 条，降低 98.4%。
- 资源：Agent CPU 降 37.5%，内存降 2.0%。
- 文献：正文引用 51 篇，归档 ok=51。
- 微基准：L0 为 0.5 ms，v9 稿件为约 15 ms，需后续统一。
