# 02 · 核心概念与技术原理

## 1. 定向采集 vs 全量采集

| 维度 | 全量采集（基线） | 定向采集（本文） |
|------|----------------|----------------|
| 触发 | 所有应用日志一律上传 | 由**关注清单**驱动，只采匹配 URL/级别 |
| 网关角色 | 仅转发 | 采集流量日志 → 驱动清单 |
| 节点开销 | 高（持续读盘 + 上传） | 低（规则过滤后批量上传） |
| Loki 压力 | 大 | 小（Center 二次过滤） |
| 信息损失 | 无 | 可控（ERROR 始终保留；空清单时仅 ERROR+） |

**核心 trade-off**：用可解释的「高价值流量信号」换资源，而非黑盒采样。

## 2. 系统角色

```
[OpenResty 网关] --流量日志--> [Redis]
                                    |
[Center] <---gRPC---> [Sidecar Agent × N] <-- 读应用日志
    |                      ^
    |                      | 关注清单 / 过滤规则
    v                      |
 [Loki]              [BoltDB 兜底]
```

| 组件 | 职责 |
|------|------|
| **Gateway Agent** | 采集 Nginx/OpenResty 流量，上传 Center |
| **Service Agent** | 按清单匹配应用日志，固定缓存块 + gRPC 上传 |
| **Center** | 清单生成、规则下发、二次过滤、写 Loki |
| **Redis** | 网关流量日志高速缓存 |
| **Prometheus/Grafana** | 运行时指标与可视化 |

## 3. 定向策略三次转换

这是本文的**算法主线**（实现见 `center/pkg/strategy/triple_transform.go`）：

| 次序 | 输入 | 输出 | 作用 |
|------|------|------|------|
| **第一次** | 定向策略（超时阈值 T、错误码） | 网关采集规则 | 控制网关采哪些流量 |
| **第二次** | 网关流量日志 + 策略 | **关注清单** → Agent 过滤规则 | 驱动各节点定向采集 |
| **第三次** | 关注清单 + Agent 上传日志 | Loki 存储规则（去重、标签） | Center 最终入库把关 |

周期性执行：`RunPeriodicGeneration` 每 interval 从 Redis 拉窗口内网关日志 → 第二、三次转换。

## 4. 关注清单（Attention List）

**生成流程**（`high_value_filter.go` + `url_cluster.go` + `attention_list.go`）：

1. 过滤高价值记录（慢请求、5xx、高错误率 URL）
2. URL 聚类/泛化（`/api/user/123` → `/api/user/*`）
3. 按权重排序，取 Top-K 模式下发

**Agent 侧匹配**（`agent/pkg/matcher/rule_matcher.go`）：

- URL 模式匹配 + 日志级别过滤
- **空清单退化策略**（三期修复）：无清单时仅采集 ERROR 及以上，避免 INFO 洪泛

## 5. 固定缓存块

每个 Agent 分配固定内存环形队列（默认 64–256 MB）：

- 超阈值：懒压缩 / 丢弃低优先级
- 与**压力感知**联动：CPU/内存高时降低上传频率
- 目的：节点资源**有上界**，避免 OOM

## 6. 指数退避重试

公式：`delay = baseDelay × multiplier^attempt + jitter`

| 参数 | 默认值 |
|------|--------|
| baseDelay | 200 ms |
| multiplier | 2.0 |
| maxDelay | 30 s |
| maxRetries | 6 |
| jitter | 0.3 |

与 BoltDB 本地兜底配合：Center 不可达时先落盘，恢复后补传。单元测试见 `agent/pkg/retry/backoff_test.go`。

## 7. 实验假设（H1–H5）

| 假设 | 内容 | 三期结论 |
|------|------|---------|
| H1 | CPU/内存降 ≥40% | CPU 降 37.5%（接近）；内存 2% |
| H2 | 带宽降 ≥60% | **98.4%**（与日志量同比例） |
| H3 | 日志量降 65–80% | **98.4%**（超出，需披露实验条件） |
| H4 | 故障丢失率 <1% | 退避单测通过；R2 待补 |
| H5 | 清单生成 <100 ms | 首期微基准 **0.5 ms**（5000 条输入） |

## 8. 与 ELK/Loki 全量栈的关系

- **不是**替换 Loki，而是替换 Promtail/全量 Agent 的采集策略
- Center 仍通过 Loki Push API 写入
- 对比基线：关闭定向过滤的「全量 Agent」配置（`experiments/results/phase3/` 中 `full_collect` 轮次）

## 9. 与 AIOps 和日志诊断研究的关系

v6 稿件将本文定位为 **AIOps 采集前端的定向减量技术**：

| 参照方向 | 代表文献/框架 | 本文关系 |
|----------|--------------|----------|
| AIOps 标准化 | 包航宇等 AIOps-OSA | 支撑其中“数据采集”能力，降低进入平台的数据规模 |
| 日志故障诊断 | 贾统等日志诊断综述 | 不替代后端解析/异常检测/RCA，而是减少其输入规模 |
| 云原生可观测性 | OpenTelemetry、Loki、eBPF、Service Mesh | 与 trace/span 采样和基础设施事件采集互补 |
