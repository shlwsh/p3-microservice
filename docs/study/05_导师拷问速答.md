## §二 RQ 速答

**Q：RQ1 怎么验证？**
A：看表1和图3。Loki 入库 72 条 vs 4388±1 条，降低 98.4%。

**Q：RQ2 怎么验证？**
A：同一主表给出 Agent CPU 和内存。CPU 降幅 37.5%，内存降幅 2.0%，说明资源开销有改善但不是主贡献。

**Q：RQ3 端到端闭环是什么？**
A：Nginx 共享内存 → Agent gRPC → Redis → Center 生成关注清单 → Agent 拉取规则 → 定向上传 → Loki。

**Q：RQ4 怎么支撑？**
A：缓存和退避由策略仿真消融、微基准、编号退避公式和 Go 单元测试共同支撑；集群实测消融仍是后续工作。

## §三 架构类

**Q：为什么用 Sidecar？**
A：Sidecar 与业务实例边界一致，便于按服务下发清单和采集本地应用日志；代价是有额外资源开销，所以论文报告 CPU/内存指标。

**Q：Center 挂了怎么办？**  
A：Agent 使用指数退避和 BoltDB 本地兜底；恢复后补传。中心故障恢复时间还需后续实测量化。

**Q：和 Promtail 的区别？**  
A：Promtail 通常基于静态标签或级别采集/过滤；本文策略来自网关流量，关注慢请求、错误码和 URL 模式，属于采集前端动态减量。

## §四 实验公平性

**Q：定向和全量实验是否完全单变量？**
A：不是。必须主动说明：定向模式应用日志发射间隔为 2 s，全量模式为 400 ms。98.4% 是策略过滤与源配置差异的联合效果。

**Q：98.4% 会不会过高？**
A：L0 三轮均为 72 条，full 为 4387/4388/4389 量级，均值 4388±1。高降幅来自多节点全量基线、空清单 ERROR+ 兜底和 URL 对齐，但不应外推到所有生产场景。

**Q：CPU 为什么只降 37.5%？**
A：Agent 本身很轻，CPU 不是主要瓶颈；本文主目标是日志入库量、带宽和存储压力下降。

## §五 算法类

**Q：关注清单怎么生成？**  
A：Redis 窗口网关日志 → 高价值过滤 → URL 泛化 → Top-K 加权排序 → 版本化清单 → gRPC 下发。

**Q：空清单怎么办？**  
A：只采 ERROR 及以上日志，避免退化为全量 INFO 洪泛。

**Q：退避为什么加 jitter？**  
A：避免多个 Agent 同时重试造成 Center 瞬时拥塞。

## §九 一句对比

| 方案 | 差异 |
|------|------|
| ELK/EFK 全量 | 采集前端不减量，存储和索引压力大 |
| Loki/Promtail | 更偏静态标签和管道配置 |
| OpenTelemetry 尾采样 | 面向 trace/span，不直接解决应用日志上传前过滤 |
| eBPF | 偏基础设施事件，应用语义需额外映射 |
| 本文 | 网关流量驱动关注清单，在 Sidecar 上传前做应用日志定向采集 |
