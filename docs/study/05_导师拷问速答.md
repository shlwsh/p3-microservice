## §二 科学问题速答

**Q：SQ1 怎么验证？**  
A：E1–E3 对比实验，以 Loki 入库条数、Agent CPU、带宽为主指标；三期实测 98.4% 日志降幅。

**Q：SQ2 固定缓存有什么用？**  
A：给每个节点内存上界；消融显示去掉缓存 CPU 从 15.2%→44.8%（仿真），丢失率 0.1%→2.5%。

**Q：SQ3 三次转换是什么？**  
A：策略→网关规则；网关日志→关注清单→Agent 规则；清单+服务日志→Loki 存储规则。见 `triple_transform.go`。

## §三 架构类

**Q：为什么 Sidecar 而不是 DaemonSet 全量？**  
A：Sidecar 与业务 Pod 同生命周期，可读取容器内日志；规则按服务实例下发，符合微服务边界。

**Q：Center 挂了怎么办？**  
A：Agent BoltDB 本地兜底 + 指数退避补传；R2 中心宕机实验待完整量化。

**Q：和 Promtail 的区别？**  
A：Promtail 按 label 全量推；我们是**流量驱动的动态清单**，只采高价值请求相关日志。

## §四 实验公平性（审核高频）

**Q：定向和全量实验条件一样吗？**  
A：**不完全相同，必须主动披露**——  
- 定向 Agent 日志发射间隔 **2 s**  
- 全量模式 **400 ms**（`collector.go`）  
- 全量模式有意维持高吞吐基线；定向侧有规则过滤 + 较低发射频率  
- 这是「能力对比」而非严格单变量；正文 §6 需说明并讨论 external validity

**Q：为什么二期 31% 三期 98%？**  
A：（1）空清单 INFO 退化 bug 已修复；（2）应用日志 URL 与网关模式对齐；（3）8 节点放大全量基线；（4）时长 45s→180s。

**Q：98.4% 会不会太高不可信？**  
A：L0 三轮均为 72 vs 4388±1，可复现；关键是合理解释实验条件（发射频率、节点数、过滤规则），而非隐瞒。

## §五 算法类

**Q：关注清单怎么生成？**  
A：Redis 窗口内网关日志 → 高价值过滤（慢/错）→ URL 聚类泛化 → Top-K 加权排序 → gRPC 下发。

**Q：空清单怎么办？**  
A：仅采集 **ERROR+**（WARN/FATAL 视配置），避免 INFO 洪泛；这是三期降幅的重要机制。

**Q：退避为什么加 jitter？**  
A：避免多 Agent 同时重试造成 Center **惊群**。

## §八 常见陷阱（答辩避坑）

| 陷阱 | 正确说法 |
|------|---------|
| 「CPU 降低 98%」 | 主指标是 **Loki 入库**，CPU 只降 37.5% |
| 「仿真实测都是 98%」 | 首期 68% 是**仿真**；98.4% 是**三期实测** |
| 「σ = 1000 说明波动大」 | σ = **1**，1000 是笔误 |
| 「消融证明 98%」 | 消融是**仿真趋势**；98% 来自三期对比 |
| 「完全无信息损失」 | ERROR 始终保留；非匹配 URL 的 INFO 会丢弃——trade-off |
| 「已生产就绪」 | 原型为 Docker Compose/WSL，K8s Helm 待建设 |
| 「参考文献还不足」 | v6 已有 **73** 篇正文引用，且全部归档到 `data/papers/` |
| 「所有实验风险已关闭」 | 文献门禁已关闭；大规模真实负载和工业基线仍未实测 |
