本文档对应论文 **§4 关键算法**，与代码实现一一映射。数字实验解读见 [03_实验数据与图表解读.md](./03_实验数据与图表解读.md)。

## 1. 高价值流量过滤

**文件**：`center/pkg/strategy/high_value_filter.go`

**输入**：网关流量日志条目（URL、status、response_time_ms、client_ip）

**规则**（可配置）：

- 响应时间 > 阈值 T（默认数百 ms）
- HTTP 状态码 ∈ 错误码列表（5xx 等）
- URL 错误率超过阈值

**输出**：高价值子集，供 URL 聚类使用。

**复杂度**：O(N) 单次扫描。

---

## 2. URL 聚类与泛化

**文件**：`center/pkg/strategy/url_cluster.go`

**目的**：将 `/api/order/1001`、`/api/order/1002` 泛化为 `/api/order/*`，减少清单条目数。

**步骤**：

1. 按路径段对齐
2. 数字/UUID 段替换为通配
3. 合并相似模式，保留权重（出现频次 × 严重度）

**输出**：候选 URL 模式列表。

## 3. 关注清单动态生成

**文件**：`center/pkg/strategy/attention_list.go`

```text
Generate(gatewayLogs):
  1. filtered = HighValueFilter(gatewayLogs)
  2. patterns = URLCluster(filtered)
  3. ranked = SortByWeight(patterns)
  4. return TopK(ranked), version, expiresAt
```

| 字段 | 含义 |
|------|------|
| `version` | 清单版本号，Agent 用于增量更新 |
| `items[].pattern` | URL 通配模式 |
| `items[].weight` | 权重（高权重 → 降低 min_level 到 INFO） |
| `expires_at` | 过期时间，过期后 Agent 重新拉取 |

**微基准**（L0 `attention_list_bench`）：5000 条输入 → **0.5 ms**。学习手册以 L0/L1 为准；若论文某版本正文与该数值不一致，下一轮改稿需统一。

## 4. 定向策略三次转换

**文件**：`center/pkg/strategy/triple_transform.go`

### 4.1 第一次转换：策略 → 网关规则

```go
// GatewayCollectionRule — Nginx Lua 可执行条件
type GatewayCollectionRule struct {
    MinResponseTimeMs int64
    ErrorCodes        []int
    SampleRate        float64
}
```

控制 OpenResty `log_capture.lua` 哪些请求写入 Redis。

### 4.2 第二次转换：网关日志 → Agent 规则

```go
func (t *TripleTransformer) SecondTransform(gatewayLogs []GatewayLog) *AgentFilterRule
```

1. `ListGenerator.Generate(gatewayLogs)` → AttentionList  
2. 构造 `AgentFilterRule{Items, MinLevel, ExpiresAt}`  
3. `Dispatcher.BroadcastRules(rule)` → 所有 Agent  

**MinLevel 逻辑**：存在 weight > 0.8 的模式时降为 INFO，否则 WARN；ERROR 始终通过。

### 4.3 第三次转换：清单 → Loki 存储规则

```go
func (t *TripleTransformer) ThirdTransform(currentList *AttentionList) *receiver.StorageRule
```

- 设置 Loki labels（`job=directed-log-collector`, `version=...`）
- 提取字段：service_name, url, status_code, trace_id, level
- 去重窗口：60 s（同一 TraceID）

### 4.4 周期性调度

```go
func (t *TripleTransformer) RunPeriodicGeneration(ctx, interval)
```

每 interval（如 30 s）：

1. `GatewayStore.FetchWindow(timeWindowSec)` 从 Redis 读网关日志  
2. `SecondTransform` + `ThirdTransform`

### 4.5 正确性边界

三次转换依赖同一清单版本号、同一 URL 泛化函数和未过期 TTL。只要服务日志能解析出同一 URL 字段，Agent 命中的日志不会在 Center 因策略语义漂移被错误丢弃；字段缺失或格式不兼容时，系统退化为 ERROR 级别兜底。

## 5. Agent 规则匹配

**文件**：`agent/pkg/matcher/rule_matcher.go`

```text
Match(log, rule):
  if log.Level >= ERROR: return true          // 始终保留错误
  if rule.Items is empty: return level >= ERROR  // 空清单退化（三期修复）
  if URL match any rule.Items.pattern: return level >= rule.MinLevel
  return false
```

**答辩要点**：空清单退化策略是三期 98.4% 的关键机制之一，须在论文中说明。

## 7. 指数退避与上传

**文件**：`agent/pkg/retry/backoff.go`, `agent/pkg/uploader/grpc_uploader.go`

**退避公式**：

```text
delay = min(maxDelay, baseDelay × multiplier^attempt) × (1 ± jitter)
```

**上传流程**：

```text
Collect → CacheBlock → UploadBatch(gRPC)
  ├─ success → 清除 BoltDB 对应条目
  └─ failure → BackoffSleep → 写 BoltDB → 重试
```

**单元测试**：`backoff_test.go` 验证单调递增、上限、jitter 边界。

## 8. 复杂度汇总

| 算法 | 时间复杂度 | 空间 |
|------|-----------|------|
| 高价值过滤 | O(N) | O(1) 额外 |
| URL 聚类/泛化 | O(N) 到 O(N log N)，取决于聚合结构 | O(N) |
| 清单 Top-K | 当前排序实现 O(N log N)；堆实现可为 O(N log K) | O(K) 到 O(N) |
| 规则匹配（单条日志） | O(P) 模式数 | O(1) |
| 固定缓存块 | O(1) 入队 | O(B) 块大小 |
