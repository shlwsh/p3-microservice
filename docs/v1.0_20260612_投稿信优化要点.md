# 投稿信（Cover Letter）优化要点

> 版本：v1.0 | 日期：2026-06-12 | 基于《软件学报》投稿实践总结

---

## 一、结构规范（五段式）

投稿信应包含以下五个段落，每段用**加粗标签**开头（如 `\textbf{研究背景}\quad`），不使用 `\subsection*`，以节省纵向空间。

| 段落 | 字数建议 | 核心内容 |
|------|----------|----------|
| **研究背景** | 80–120 字 | 领域痛点 → 传统方法不足 → 本研究的切入点 |
| **主要贡献** | 200–300 字 | 编号分列，每项含"做了什么+解决了什么+和已有的区别" |
| **实验验证** | 80–120 字 | 实验设置（规模、重复次数）、关键量化结果、可复现性 |
| **期刊契合** | 40–60 字 | 属于哪个方向 + 与期刊定位的关系 |
| **声明** | 40–60 字 | 原创、未发表、未一稿多投、作者同意、感谢 |

## 二、篇幅控制（一页 A4）

投稿信**必须控制在一页 A4 纸内**。实现手段：

1. **字号**：11pt（不用 12pt）。
2. **页边距**：上下 2.2cm，左右 2.5cm。
3. **段间距**：`\parskip=0.3em`，不使用 `\vspace` 分隔主体段落。
4. **贡献分列**：每项贡献独立成段但不留空行之外的额外间距；使用行内编号（1）（2）（3）（4），不用 `enumerate` 环境。
5. **签名区**：合并为两行 tabular，不要多余空行。
6. **如果超出一页**：优先压缩"研究背景"和"期刊契合"，贡献和实验数据不删。

## 三、中文润色要点

### 3.1 去模板化

| 禁用表达 | 替换方式 |
|----------|----------|
| 综上所述 | 从上述分析可以看出 / 直接省略 |
| 此外 | 同时 / 另外 / 直接省略 |
| 值得注意的是 | 删除，直接说 |
| 形成本质区别 | 有本质不同 |
| 制约了…效率 | 拖慢了…效率 |
| 造成…显著压力 | 让…都吃紧 |

### 3.2 短句优先

- 一句话不超过 30 个汉字（不含引用标记和公式）。
- 长句拆分方法：按"主语—谓语—宾语"拆成 2–3 个短句，用逗号或句号隔开。
- 列举超过 3 项时用分号隔开，不要一逗到底。

### 3.3 动词选择

- 用"把""让""跟着""走"等日常动词替代"实现""进行""开展"等官腔动词。
- 保留"提出""设计""构建""验证"等学术核心动词。
- 示例：
  - ❌ "实现采集策略的运行时自适应"
  - ✅ "采集策略跟着实际流量走"

### 3.4 保持学术自信

- 不用"尝试""初步探索"等弱化词，用"提出""设计""解决"。
- 数据表述保持精确：`降幅 98.4%`，不说"大幅降低"。
- 可以用"有效""显著"等词，但必须有数据支撑。

## 四、贡献分列写法

每项贡献遵循三段式：**做了什么 → 怎么做的 → 和已有方案的区别**。

```
（N）**贡献名称**。一句话说做了什么。
接着用一到两句说具体怎么做的（技术手段）。
最后一句点出和已有方案的区别或优势。
```

**示例**：

> （3）**定向策略三次转换算法**。用统一的 URL 泛化函数和清单版本号，让同一份策略在三层之间保持语义一致，避免跨层漂移。这和 Envoy、Kong 等只在单个节点做日志过滤的做法有本质不同。

## 五、版本化输出

投稿信编译后输出到 `docs/` 目录时，文件名必须包含版本号和时间戳：

```
docs/v<版本号>-投稿信-<YYYYMMDD-HHMMSS>.pdf
```

- 版本号从 `docs/` 已有文件自动检测最大值。
- 与论文稿件版本号保持一致或递增。
- 旧的无版本号文件（如 `投稿信.pdf`）应清理。

## 六、检查清单

编译前逐项确认：

- [ ] 全文是否控制在一页 A4 纸内？
- [ ] 五个段落是否齐全（背景、贡献、验证、契合、声明）？
- [ ] 贡献是否分列为独立段落、每项有加粗标题？
- [ ] 关键实验数据（降幅、CPU、节点数、重复次数）是否完整？
- [ ] 是否消除了"综上所述""此外""值得注意的是"等模板词？
- [ ] 每句话是否不超过 30 个汉字？
- [ ] 通讯作者、单位、邮箱、日期是否正确？
- [ ] 输出文件名是否带版本号和时间戳？
