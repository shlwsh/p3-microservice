# 微服务架构下分布式定向日志采集组件 — 实现方案

> 基于 [v1-设计方案.md](./v1-设计方案.md) 的论文设计，全部采用 **Go 语言**实现，存储后端采用 **Grafana Loki**。

---

## 1 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| 节点采集器（Agent） | Go 1.22 | 轻量高并发，适合 Sidecar 部署 |
| 日志中心（Center） | Go 1.22 | 统一技术栈，降低维护成本 |
| 通信协议 | gRPC + Protobuf | 高性能二进制传输，支持双向流 |
| 高速缓存 | Redis | 缓存网关流量日志 + 关注清单 |
| 日志存储 | Grafana Loki | 基于标签的日志存储，轻量高效 |
| 可视化 | Grafana | 原生支持 Loki 数据源 |
| 指标监控 | Prometheus | Agent/Center 运行时指标 |
| 本地兜底 | BoltDB | 嵌入式 KV，无外部依赖 |
| 网关采集 | OpenResty + Lua | Nginx `log_by_lua` 阶段采集流量 |
| 容器化 | Docker + Kubernetes | Helm Chart 部署 |

## 3 核心算法实现

### 3.1 指数退避重试机制

**文件**：`agent/pkg/retry/backoff.go`

**公式**：
```
delay = baseDelay × multiplier^attempt + rand(0, expDelay × jitter)
clamp(delay, 0, maxDelay)
```

**参数**（论文推荐值）：

| 参数 | 值 | 说明 |
|------|----|------|
| baseDelay | 200ms | 基础延迟 |
| multiplier | 2.0 | 指数增长因子 |
| maxDelay | 30s | 最大延迟上限 |
| maxRetries | 6 | 最大重试次数 |
| jitter | 0.3 | 抖动因子（防惊群） |

**关键实现**：
- `NextDelay(attempt)` — 计算第 N 次重试延迟
- `NextDelayWithPressure(attempt, isHighPressure)` — 高压模式延迟翻倍
- 不可重试错误（认证失败等）立即放弃 → BoltDB 本地兜底
- 中心恢复后自动补传

### 3.2 固定缓存块算法

**文件**：`agent/pkg/cache/ring_buffer.go` + `cache_block.go`

- 基于 `container/ring` 的环形队列
- 双重容量约束：条目数 + 内存字节数
- 满时覆盖最旧条目（FIFO 淘汰）
- 异步上传触发：容量阈值 80% 或定时器（5s）
- gzip 压缩：减少网络传输量

### 3.3 定向规则匹配引擎

**文件**：`agent/pkg/matcher/rule_matcher.go`

- 支持 4 种匹配模式：exact / prefix / regex / wildcard
- 关键词匹配（大小写不敏感）
- 权重排序：高优先级规则优先匹配
- **无锁热更新**：`atomic.Value` 原子替换规则集

### 3.4 关注清单动态生成算法

**文件**：`center/pkg/strategy/attention_list.go`

**算法流程**（O(N log N)）：
1. **高价值过滤** — 筛选慢请求（> T ms）、错误码（5xx）、高错误率 URL
2. **URL 聚类泛化** — `/api/user/123` → `/api/user/{id}`
3. **权重计算** — `weight = countFactor × severityScore`
4. **Top-K 筛选** — 保留权重最高的 K 个模式
5. 构建清单 + 附加 TTL → 下发至各 Agent

### 3.5 URL 聚类/泛化算法

**文件**：`center/pkg/strategy/url_cluster.go`

- 纯数字段 → `{id}`
- UUID 格式段 → `{uuid}`
- 长十六进制串 → `{hex_id}`（如 MongoDB ObjectId）
- 查询参数泛化：保留键名，移除值

### 3.6 定向策略三次转换算法

**文件**：`center/pkg/strategy/triple_transform.go`

| 转换阶段 | 输入 | 输出 | 作用 |
|---------|------|------|------|
| 第一次 | 定向策略 | 网关采集规则 | Nginx Lua 预筛选（阈值 T/5） |
| 第二次 | 网关日志 + 策略 | 关注清单 → Agent 规则 | 驱动各节点定向采集 |
| 第三次 | 关注清单 + 服务日志 | Loki 存储规则 | 二次过滤 + 结构化入库 |

## 4 Loki 存储集成

**文件**：`center/pkg/storage/loki.go`

### 4.1 数据模型

Loki 以 **标签（labels）** + **时间戳** + **日志行（line）** 为核心模型：

```json
{
  "streams": [{
    "stream": {
      "job": "directed-log-collector",
      "service": "user-service",
      "level": "ERROR",
      "source": "APPLICATION"
    },
    "values": [
      ["1717567890000000000", "{\"url\":\"/api/user/123\",\"status\":500,...}"]
    ]
  }]
}
```

### 4.2 关键实现

- **批量推送**：缓存日志条目，达到 `batch_size`（100）或 `batch_wait`（1s）后推送
- **gzip 压缩**：通过 `Content-Encoding: gzip` 减少网络传输
- **标签映射**：`service_name` → `service` label, `level` → `level` label
- **重试机制**：Loki 5xx 错误时重试（最多 3 次）
- **多租户**：通过 `X-Scope-OrgID` 头支持 Loki 多租户

## 6 实验方案概要

### 6.1 对比实验

| 编号 | 对比项 | 指标 | 预期结论 |
|------|--------|------|---------|
| E1 | 定向 vs Loki 全量 | CPU/内存 | 定向降低 ~60% |
| E2 | 定向 vs Loki 全量 | 网络带宽 | 定向降低 70%+ |
| E3 | 定向 vs Loki 全量 | 日志量 | 定向减少 65%-80% |
| E4 | 不同并发量 | 延迟/吞吐 | 高并发优势放大 |

### 6.2 消融实验

| 编号 | 移除模块 | 预期影响 |
|------|---------|---------|
| A1 | 动态清单 | 日志量回升至全量 |
| A2 | 固定缓存块 | 内存飙升、OOM |
| A3 | 指数退避 | 惊群、丢失率↑ |
| A4 | 压力感知 | 高负载不稳定 |

### 6.3 重试机制验证

| 场景 | 模拟方式 | 关注指标 |
|------|---------|---------|
| 网络抖动 | tc netem | 成功率、P99 延迟 |
| 中心宕机 | kill 进程 | 本地兜底量 |
| 持续高压 | CPU quota | 退避自适应 |
