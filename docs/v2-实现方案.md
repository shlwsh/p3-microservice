# 微服务架构下分布式定向日志采集组件 — 实现方案

> 基于 [v1-设计方案.md](./v1-设计方案.md) 的论文设计，全部采用 **Go 语言**实现，存储后端采用 **Grafana Loki**。  
> **部署目标**：以 **WSL2 + Docker Compose** 作为本地开发与实验环境，Kubernetes 作为生产部署目标。

---

## 1 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| 节点采集器（Agent） | Go 1.22 | 轻量高并发，适合 Sidecar 部署 |
| 日志中心（Center） | Go 1.22 | 统一技术栈，降低维护成本 |
| 通信协议 | gRPC + Protobuf | 高性能二进制传输，支持双向流 |

## 3 核心算法实现

### 3.1 指数退避重试机制

**文件**：`agent/pkg/retry/backoff.go`

**公式**：
```
delay = baseDelay × multiplier^attempt + rand(0, expDelay × jitter)
clamp(delay, 0, maxDelay)
```

**参数**（论文推荐值）：

| 参数 | 值 | 说明 |
|------|----|------|
| baseDelay | 200ms | 基础延迟 |
| multiplier | 2.0 | 指数增长因子 |
| maxDelay | 30s | 最大延迟上限 |
| maxRetries | 6 | 最大重试次数 |
| jitter | 0.3 | 抖动因子（防惊群） |

**关键实现**：
- `NextDelay(attempt)` — 计算第 N 次重试延迟
- `NextDelayWithPressure(attempt, isHighPressure)` — 高压模式延迟翻倍
- 不可重试错误（认证失败等）立即放弃 → BoltDB 本地兜底
- 中心恢复后自动补传

### 3.2 固定缓存块算法

**文件**：`agent/pkg/cache/ring_buffer.go` + `cache_block.go`

- 基于 `container/ring` 的环形队列
- 双重容量约束：条目数 + 内存字节数
- 满时覆盖最旧条目（FIFO 淘汰）
- 异步上传触发：容量阈值 80% 或定时器（5s）
- gzip 压缩：减少网络传输量

### 3.3 定向规则匹配引擎

**文件**：`agent/pkg/matcher/rule_matcher.go`

- 支持 4 种匹配模式：exact / prefix / regex / wildcard
- 关键词匹配（大小写不敏感）
- 权重排序：高优先级规则优先匹配
- **无锁热更新**：`atomic.Value` 原子替换规则集

### 3.4 关注清单动态生成算法

**文件**：`center/pkg/strategy/attention_list.go`

**算法流程**（O(N log N)）：
1. **高价值过滤** — 筛选慢请求（> T ms）、错误码（5xx）、高错误率 URL
2. **URL 聚类泛化** — `/api/user/123` → `/api/user/{id}`
3. **权重计算** — `weight = countFactor × severityScore`
4. **Top-K 筛选** — 保留权重最高的 K 个模式
5. 构建清单 + 附加 TTL → 下发至各 Agent

### 3.5 URL 聚类/泛化算法

**文件**：`center/pkg/strategy/url_cluster.go`

- 纯数字段 → `{id}`
- UUID 格式段 → `{uuid}`
- 长十六进制串 → `{hex_id}`（如 MongoDB ObjectId）
- 查询参数泛化：保留键名，移除值

### 3.6 定向策略三次转换算法

**文件**：`center/pkg/strategy/triple_transform.go`

| 转换阶段 | 输入 | 输出 | 作用 |
|---------|------|------|------|
| 第一次 | 定向策略 | 网关采集规则 | Nginx Lua 预筛选（阈值 T/5） |
| 第二次 | 网关日志 + 策略 | 关注清单 → Agent 规则 | 驱动各节点定向采集 |
| 第三次 | 关注清单 + 服务日志 | Loki 存储规则 | 二次过滤 + 结构化入库 |

## 4 Loki 存储集成

**文件**：`center/pkg/storage/loki.go`

### 4.1 数据模型

Loki 以 **标签（labels）** + **时间戳** + **日志行（line）** 为核心模型：

```json
{
  "streams": [{
    "stream": {
      "job": "directed-log-collector",
      "service": "user-service",
      "level": "ERROR",
      "source": "APPLICATION"
    },
    "values": [
      ["1717567890000000000", "{\"url\":\"/api/user/123\",\"status\":500,...}"]
    ]
  }]
}
```

### 4.2 关键实现

- **批量推送**：缓存日志条目，达到 `batch_size`（100）或 `batch_wait`（1s）后推送
- **gzip 压缩**：通过 `Content-Encoding: gzip` 减少网络传输
- **标签映射**：`service_name` → `service` label, `level` → `level` label
- **重试机制**：Loki 5xx 错误时重试（最多 3 次）
- **多租户**：通过 `X-Scope-OrgID` 头支持 Loki 多租户

#### 端口映射

| 服务 | 宿主机端口 | 容器端口 | 用途 |
|------|-----------|---------|------|
| Gateway (Nginx) | **8088** | 80 | HTTP 网关入口（WSL 下 80 常被 Windows IIS 占用） |
| Center HTTP API | 8080 | 8080 | 管理 API / 健康检查 |
| Center gRPC | **9091** | 9090 | Agent 通信（避免与 Prometheus 冲突） |
| Loki | 3100 | 3100 | 日志存储 API |
| Grafana | 3000 | 3000 | 可视化面板 |
| Redis | 6379 | 6379 | 高速缓存 |
| Prometheus | 9090 | 9090 | 指标采集 |
| Agent Metrics | 9100 | 9100 | Sidecar 指标（经 network_mode 共享） |

> **注意**：Center gRPC 对外映射为 `9091`，因 Prometheus 已占用 `9090`。容器内部 Agent 仍通过 `center:9090` 通信，不受影响。

#### 配置分层说明

| 配置文件 | 使用场景 | 地址风格 |
|---------|---------|---------|
| `agent/configs/agent.yaml` | 本地 `go run` 调试 | `localhost:9090` |
| `center/configs/center.yaml` | 本地 `go run` 调试 | `localhost:6379` |
| `deploy/docker/center-config.yaml` | Docker 容器 | `redis:6379`, `loki:3100` |
| `deploy/docker/gateway-agent-config.yaml` | 网关 Sidecar | `center:9090` |
| `deploy/docker/service-agent-config.yaml` | 服务 Sidecar | `center:9090` |

Docker 网络内服务通过 **容器名** 互访（如 `center`、`redis`、`loki`），宿主机通过 **localhost + 映射端口** 访问。

#### Sidecar 网络模式

Agent 使用 `network_mode: "service:gateway"` 与业务容器共享网络栈：

- Agent 可直接访问同容器内的 Nginx 共享内存（`/dev/shm/nginx_log`）
- Prometheus 通过 gateway 容器的 `9100` 端口采集 Agent 指标
- 业务流量经 gateway 代理至 `service-1` / `service-2`

### 5.3 WSL 混合开发模式

适用于修改 Go 代码后快速迭代，无需每次重建 Docker 镜像。

**步骤 1：仅启动基础设施**

```bash
cd deploy/docker
docker compose up -d loki grafana redis prometheus
```

**步骤 2：本地运行 Center**

```bash
cd center
export CENTER_CONFIG_PATH=./configs/center.yaml
go run ./cmd/center/
```

**步骤 3：本地运行 Agent**

```bash
cd agent
export AGENT_CONFIG_PATH=./configs/agent.yaml
go run ./cmd/agent/
```

此时 `configs/*.yaml` 中的 `localhost` 地址指向 WSL 宿主机上映射的 Docker 端口，可直接联通。

### 5.5 WSL 常见问题排查

| 现象 | 原因 | 解决方案 |
|------|------|---------|
| `docker: command not found` | 未安装或未启用 WSL 集成 | 安装 Docker Desktop，Settings → Resources → WSL Integration 勾选当前发行版 |
| `Cannot connect to the Docker daemon` | Docker Desktop 未启动 | 启动 Docker Desktop，等待托盘图标就绪 |
| 端口被占用（8088/3000/9090） | 本机已有服务监听 | `ss -tlnp \| grep :8088` 查找进程并停止，或修改 compose 端口映射 |
| `localhost:80` 返回 IIS 404 | Windows IIS 占用 80 端口 | 使用 `http://localhost:8088` 访问网关（已默认配置） |
| `docker-credential-desktop.exe: exec format error` | WSL 凭证助手不兼容 | `setup.sh` 会自动修复；或手动设置 `~/.docker/config.json` 为 `{"auths":{},"credsStore":""}` |
| Gateway 容器 Exited (1) | Nginx 日志目录不存在 | 已改为输出到 stdout/stderr；若仍失败执行 `docker compose up -d --force-recreate gateway` |
| 构建极慢 / 磁盘 I/O 高 | 代码在 `/mnt/c/` | 将项目移至 `~/work/` 等 WSL 原生路径 |
| 容器 OOM / 频繁重启 | WSL 内存不足 | 增大 `.wslconfig` memory，或使用 `docker-compose.wsl.yml` |
| Grafana 无法查日志 | 未配置 Loki 数据源 | Grafana → Connections → Data Sources → 添加 Loki `http://loki:3100` |
| Agent 连不上 Center | 配置地址错误 | 容器内用 `center:9090`，本地调试用 `localhost:9091`（gRPC 映射端口） |
| `network_mode: service:gateway` 启动失败 | gateway 未就绪 | 确保 `depends_on: center` 链路正常，增加 `setup.sh` 中的等待时间 |

### 5.6 资源预算（WSL 8GB 内存）

启用 `docker-compose.wsl.yml` 后，各服务内存上限合计约 **2 GB**，剩余空间留给 WSL 系统、IDE 和编译缓存：

| 服务 | mem_limit |
|------|-----------|
| Loki | 512 MB |
| Grafana | 256 MB |
| Redis | 192 MB |
| Prometheus | 256 MB |
| Center | 256 MB |
| 3 × Agent | 各 128 MB |

内存充裕时可执行 `./deploy/wsl/setup.sh start-full` 跳过资源限制。

## 6 实验方案概要

### 6.1 对比实验

| 编号 | 对比项 | 指标 | 预期结论 |
|------|--------|------|---------|
| E1 | 定向 vs Loki 全量 | CPU/内存 | 定向降低 ~60% |
| E2 | 定向 vs Loki 全量 | 网络带宽 | 定向降低 70%+ |
| E3 | 定向 vs Loki 全量 | 日志量 | 定向减少 65%-80% |
| E4 | 不同并发量 | 延迟/吞吐 | 高并发优势放大 |

### 6.2 消融实验

| 编号 | 移除模块 | 预期影响 |
|------|---------|---------|
| A1 | 动态清单 | 日志量回升至全量 |
| A2 | 固定缓存块 | 内存飙升、OOM |
| A3 | 指数退避 | 惊群、丢失率↑ |
| A4 | 压力感知 | 高负载不稳定 |

### 6.3 重试机制验证

| 场景 | 模拟方式 | 关注指标 |
|------|---------|---------|
| 网络抖动 | `tc netem`（需 `sudo`） | 成功率、P99 延迟 |
| 中心宕机 | `docker stop log-center` | 本地兜底量 |
| 持续高压 | `docker update --cpus` | 退避自适应 |

### 6.4 WSL 实验注意事项

- **前置条件**：先执行 `./deploy/wsl/setup.sh start` 确保集群就绪
- **网络模拟**：WSL2 下 `tc netem` 需 root 权限，且作用于 `eth0` 接口；建议在 Docker 网络 namespace 内操作
- **负载工具**：需预先安装 `jmeter` 或 `wrk`；小规模验证可用 `ab` / `curl` 循环
- **指标采集**：Prometheus 地址 `http://localhost:9090`，实验脚本默认从此处拉取
- **资源监控**：实验期间建议关闭 Windows 侧大型应用，避免 WSL 内存被回收（`autoMemoryReclaim`）
